# Vault

Source: `pulumi/core/security/vault`

Vault is the cluster's runtime secrets service. In this repo, that means it is
not the place where Pulumi gets every value it needs before a deployment can
exist. Pulumi secrets and ESC-style configuration are deploy-time inputs.
Kubernetes Secrets are the cluster-native objects that workloads can mount or
read. Vault is the service you bring online after the infrastructure exists so
humans and applications can ask for secrets through an API with authentication,
authorization, auditability, and rotation patterns.

That distinction matters because Vault is powerful but not magic. Moving a
password from a Kubernetes Secret into Vault is only an improvement when the
consumer has a safe way to authenticate to Vault, a narrow policy, a known
secret path, an ownership model, and a recovery process. A value that is needed
to deploy the cluster still belongs in Pulumi/ESC or another deploy-time secret
source. A value that a running service should fetch, rotate, or receive through
a central policy can be a good Vault candidate.

This stack deploys a small standalone Vault server inside the cluster. It uses
the HashiCorp Helm chart, enables TLS and the Vault UI, exposes the service
privately through Tailscale annotations, and stores Vault data with the chart's
standalone file-storage setup at `/vault/data`. It is intentionally not a
multi-node HA Vault cluster. Treat it as an in-cluster secrets service with a
clear operational lifecycle, not as a fully managed external Vault platform.

## What This Stack Creates

The Pulumi project is intentionally compact. `Pulumi.yaml` declares a Python
3.12 project named `vault` using the `uv` toolchain. The project dependencies
are `pulumi`, `pulumi-kubernetes`, and `pulumi-tls`. The deployment behavior is
in `__main__.py`; there is no stack-specific config read by the program at the
time of this page.

Pulumi creates a Kubernetes namespace named `vault`, then creates a small
private PKI for the Vault server:

- an ECDSA P-256 CA private key
- a self-signed CA certificate with common name `vault-ca`
- an ECDSA P-256 server private key
- a server certificate signed by that CA
- a Kubernetes Secret named `vault` in the `vault` namespace containing
  `vault.key`, `vault.crt`, and `vault.ca`

The server certificate is issued for the names the service is expected to use
inside the cluster:

```text
vault
vault.vault
vault.vault.svc
vault.vault.svc.cluster.local
127.0.0.1
```

The CA certificate is configured for a long validity window. The server
certificate is configured for a shorter validity window and early renewal. The
exact certificate material is generated by Pulumi resources and stored in
Pulumi state and a Kubernetes Secret, not in git. Anyone who can read Pulumi
state or the Kubernetes Secret can read sensitive key material, so treat both
access paths as privileged.

After the TLS Secret exists, Pulumi installs the HashiCorp `vault` chart from
`https://helm.releases.hashicorp.com` at chart version `0.32.0`. The chart is
installed into the `vault` namespace with a blank Pulumi `resource_prefix`, so
Kubernetes object names stay close to the chart's native names.

The important Helm values are:

```text
global.tlsDisable:          false
injector.enabled:           false
server.standalone.enabled:  true
server.service.annotations: tailscale.com/expose=true
server.service.annotations: tailscale.com/hostname=vault
server.extraEnvironmentVars:
  VAULT_CACERT=/etc/pki/vault/vault.ca
```

The server mounts the Kubernetes Secret at `/etc/pki` and the Vault config
points the listener at those files:

```hcl
listener "tcp" {
  address            = "[::]:8200"
  cluster_address    = "[::]:8201"
  tls_cert_file      = "/etc/pki/vault/vault.crt"
  tls_key_file       = "/etc/pki/vault/vault.key"
  tls_client_ca_file = "/etc/pki/vault/vault.ca"
}

storage "file" {
  path = "/vault/data"
}
```

The same config enables the Vault UI with `ui = true`. The UI is only an
interface to Vault. It is not an auth model, and it does not make root tokens,
unseal keys, or broad policies safe to share.

The stack sets `pulumi.com/skipAwait` on Vault's server resources and service.
This is deliberate. A Vault pod can be deployed successfully while the Vault
application is still uninitialized or sealed. Kubernetes readiness, Helm
completion, and Vault usability are related signals, but they are not the same
thing. Do not remove the skip-await annotations unless a preview and live
behavior show that Pulumi can wait correctly without getting stuck on Vault's
application lifecycle.

The chart resource also has an alias for the older Helm v3 Pulumi Chart type.
That alias is state-preservation glue. Leave it alone unless you are doing an
intentional state migration.

## Deployment Is Only The First Step

Vault has two lifecycles. The infrastructure lifecycle gets the namespace,
Secret, StatefulSet, Service, and Tailscale exposure into Kubernetes. The Vault
application lifecycle starts after the server is reachable.

At a high level, a new Vault instance goes through this sequence:

```text
Kubernetes resources are created
Vault server starts
Vault is initialized
unseal key shares and the initial root token are produced
Vault is unsealed
an operator authenticates
auth methods are enabled
policies are written
secrets engines are enabled
applications and humans receive narrow access
```

Pulumi handles the first two lines. It does not initialize Vault, store unseal
key shares, store the initial root token, enable auth methods, create policies,
or load runtime secrets. That is a feature, not a missing variable. The most
sensitive Vault material should not be written into this repo, docs, commit
messages, pull requests, chat, or long-lived shell history.

Initialization is the moment when Vault creates its encryption barrier and
prints the recovery material for a fresh storage backend. For the default
Shamir-style seal, initialization returns unseal key shares and an initial root
token. The shares are used to unseal the server; the root token is used only to
bootstrap real auth and policy structure. Both are highly sensitive.

Unsealing is the process that lets a running Vault server decrypt its storage
barrier and serve requests. A sealed Vault can be healthy from Kubernetes'
point of view and still refuse secret operations. Because this stack does not
configure an auto-unseal seal stanza, expect manual operational unseal after
initialization and after restarts unless an out-of-band process has been added.

Authentication answers "who is this caller?" Authorization answers "what can
this caller do?" Vault keeps those separate. An auth method such as Kubernetes,
OIDC, AppRole, userpass, or tokens establishes identity and creates a Vault
token. A policy attached to that token says which paths and capabilities are
allowed. A secrets engine such as KV, database, PKI, or transit owns the paths
that callers interact with.

The root token should not become the integration pattern. Use it to create the
first admin path, enable real auth methods, and then put it away according to
the operational process. Applications should use an auth method and a narrow
policy. Humans should use an auth method that fits the trust model. Long-lived
shared tokens are easy to start with and difficult to clean up later.

## Storage And Durability

This stack configures Vault's `file` storage backend:

```hcl
storage "file" {
  path = "/vault/data"
}
```

In standalone chart mode, Vault runs as a single server and writes its data to
the chart-managed data path. That makes the persistent volume and the Vault
storage directory operationally important. A pod restart is not supposed to
erase Vault. Deleting the data volume, changing the storage path without a
migration, or reinitializing against an existing data directory can turn a
routine change into data loss.

Keep these boundaries clear:

- Restarting the pod is not the same as resetting Vault.
- Sealed is not the same as empty.
- Uninitialized is not the same as unreachable.
- A PVC backup is sensitive even though Vault encrypts stored secret data.
- Unseal key shares and root tokens are separate from the storage volume and
  need their own protected handling.

File storage is appropriate for a small standalone deployment, but it is not
the same as integrated Raft HA, external cloud KMS auto-unseal, or a managed
Vault service. If this stack becomes a dependency for many applications, the
next design discussion should be about HA, recovery time, backups, auto-unseal,
and auth ownership before adding more secret paths.

## TLS And Access

TLS is enabled at the Vault listener. The chart's `global.tlsDisable` is set to
`false`, and the listener uses the generated cert, key, and CA files mounted
from the Kubernetes Secret. The `VAULT_CACERT` environment variable points the
Vault process and CLI inside the pod at `/etc/pki/vault/vault.ca`.

Clients also need to trust that CA. Without the CA, the usual error is an
unknown authority or certificate verification failure. With the wrong hostname,
the usual error is a certificate name mismatch. The current certificate covers
the in-cluster service names and `127.0.0.1`, which is useful for local
port-forward debugging.

The service is annotated for Tailscale exposure:

```text
tailscale.com/expose=true
tailscale.com/hostname=vault
```

That means access is intended to stay private to the tailnet path managed by
the Tailscale operator, not through a public LoadBalancer or internet-facing
ingress. If you change the hostname, public exposure model, or TLS names, plan
for client trust and certificate SANs at the same time. A service that is
reachable through a name not present in the certificate will fail strict TLS
verification even when Vault itself is healthy.

The UI being enabled does not weaken the need for TLS and auth. Treat UI access
like API access: the browser must trust the CA, the user must authenticate, and
the token must have an appropriate policy.

## Safe Status Checks

These checks are designed to answer "what layer is broken?" without printing
secret values.

Start with Kubernetes shape:

```bash
kubectl get pods,svc,statefulset,pvc,endpoints -n vault
kubectl get events -n vault --sort-by=.lastTimestamp
kubectl logs -n vault -l app.kubernetes.io/name=vault --tail=200
```

Check that the TLS Secret exists and has the expected key names without
printing the key material:

```bash
kubectl get secret -n vault vault -o json | jq -r '.data | keys[]'
```

It is safe to inspect the public certificate metadata. Do not inspect or paste
the private key.

```bash
kubectl get secret -n vault vault -o jsonpath='{.data.vault\.crt}' \
  | base64 --decode \
  | openssl x509 -noout -subject -issuer -dates -ext subjectAltName
```

For a local CLI status check, export the public CA certificate and port-forward
to the service. The certificate includes `127.0.0.1`, so this path can verify
TLS without disabling certificate checks:

```bash
kubectl get secret -n vault vault -o jsonpath='{.data.vault\.ca}' \
  | base64 --decode > /tmp/vault.ca

kubectl -n vault port-forward svc/vault 8200:8200
```

In another terminal:

```bash
export VAULT_ADDR=https://127.0.0.1:8200
export VAULT_CACERT=/tmp/vault.ca
vault status
vault operator init -status
```

`vault status` is useful because it separates several states: unreachable,
uninitialized, sealed, and unsealed. `vault operator init -status` checks
initialization status without printing init material.

If you are inside the Vault pod, the chart already sets `VAULT_CACERT`:

```bash
kubectl exec -n vault vault-0 -- vault status
```

Do not use `-tls-skip-verify` as a routine workaround. It can help isolate a
TLS trust problem, but the durable fix is to use the right CA and hostname.

## Initialization And Unseal At A Safe Level

Only initialize an empty Vault once, and only from a prepared operational
process that has a safe place for the output before the command is run.
Initialization prints material that cannot be recovered later from this repo.
Do not paste an init transcript into an issue, doc page, terminal scrollback
shared with others, or chat.

A safe operator flow looks like this conceptually:

```text
confirm the Vault storage is new and intentionally empty
confirm the service is reachable over TLS
run initialization once
store unseal key shares according to the recovery policy
store the initial root token according to the bootstrap policy
unseal Vault
log in with the bootstrap token
create normal admin/auth/policy structure
stop using the root token for routine work
```

For unseal, prefer the interactive prompt so key shares are not placed on the
command line:

```bash
vault operator unseal
```

For login, prefer the interactive prompt for the same reason:

```bash
vault login
```

Reinitialization is not a repair action. If Vault was initialized before and
now appears empty, stop and inspect the storage path, PVC, pod events, and
recent changes. Running init again against the wrong storage state can create a
new Vault with new recovery material while the real problem remains
unresolved.

## Auth Methods, Policies, And Secret Paths

This Pulumi stack does not enable any auth methods or write any policies. That
keeps the infrastructure deployment separate from the runtime security model.
When a real consumer is added, the consumer design should name four things:

- the auth method
- the Vault policy name
- the secrets engine and path layout
- the token lifetime and rotation expectation

For applications running in Kubernetes, the usual pattern is Kubernetes auth:
Vault verifies a pod's service account identity and maps a namespace/service
account pair to a Vault role. That role attaches one or more policies. The pod
gets a Vault token with limited capabilities and a limited lifetime.

The injector is disabled in this stack, so pods are not automatically mutated
to receive Vault sidecars or injected files. Applications need an explicit
client, init container, external-secrets style controller, or another
integration that you deliberately add. Enabling the injector later is a
behavioral change for the cluster and should be reviewed like one.

Policies should be path-shaped and capability-shaped. For example, a read-only
KV v2 application policy usually needs access to the data path, and sometimes
list access to metadata:

```hcl
path "secret/data/<app>/*" {
  capabilities = ["read"]
}

path "secret/metadata/<app>/*" {
  capabilities = ["list"]
}
```

That example is intentionally generic. Do not copy real application names,
secret names, database usernames, tokens, or private infrastructure details into
public docs. The important idea is that each application gets the smallest
policy that matches how it reads or writes secrets.

A good policy review asks:

- Can this token read only the app's own paths?
- Does it need write access, or is read enough?
- Does it need `list`, or can it fetch exact paths?
- How long should tokens live?
- What happens when the service account, namespace, or app name changes?
- Who rotates the underlying secret value?

Dynamic secrets engines, such as database credentials, add another layer. The
policy may let an app request short-lived credentials instead of reading a
static password. That is often better, but only if the database role, TTL, and
revocation behavior are tested. Start with the full lifecycle, not only the
first successful read.

## Debugging Without Leaking Secrets

Debug from the outside inward. First decide whether the issue is reachability,
TLS, initialization, seal state, auth, policy, or storage. These states can
produce similar application errors, but they need different fixes.

If the service cannot be reached, check the Service, endpoints, pod, and
Tailscale exposure. A healthy-looking Service with no endpoints points at
Kubernetes selection/readiness, not Vault policy.

If TLS fails, inspect the CA, certificate dates, SANs, and the hostname the
client is using. Do not rotate the CA just because one client does not trust it.
First prove whether the client has the CA and whether the address matches a
certificate name.

If Vault says it is sealed, unseal is the missing step. Changing policies,
restarting pods, or editing Helm values will not make a sealed Vault serve
secrets.

If Vault says it is uninitialized, confirm whether this is truly a fresh
deployment. For an existing service, uninitialized status usually means the
server is looking at empty or wrong storage. Inspect PVCs and recent storage
changes before doing anything irreversible.

If Vault returns permission denied, the server is reachable and the token is
not authorized for the requested path or capability. Inspect the auth role,
attached policies, secret engine path, and token TTL. Avoid dumping complete
tokens or policy sets into shared logs.

If a pod restart loses data, focus on the data volume and storage path. The
Vault config uses `/vault/data`; the chart's volume behavior must preserve that
path. Do not use reinitialization as a substitute for storage recovery.

When sharing diagnostics, safe summaries usually include pod phase, event
reasons, certificate subject/SAN/date metadata, Vault initialized/sealed status,
and redacted error classes. Do not share:

- unseal key shares
- recovery keys
- root tokens
- client tokens
- Kubernetes Secret YAML containing `data`
- Pulumi secret outputs
- `pulumi stack output --show-secrets`
- Vault audit records or request bodies without review
- secret paths that reveal private systems or credentials

## Safe Changes

For code changes to this stack, stay close to the existing ownership boundary:
the Vault Pulumi project owns the namespace, generated TLS Secret, Helm chart,
standalone server config, and private service exposure. It does not currently
own runtime auth methods, policies, secret engines, or secret values.

Use the repository workflow for static validation and preview:

```bash
just sync pulumi/core/security/vault
just check-python
just lint
git diff --check
just preview pulumi/core/security/vault stack=mx
```

Do not run `pulumi up`, `pulumi destroy`, or `just up` unless the apply or
destructive action was explicitly requested.

Chart upgrades should be treated as migrations. Read the HashiCorp chart notes,
compare values, and preview the rendered resource changes. Pay attention to
StatefulSet changes, Service selectors, PVC behavior, and chart defaults around
standalone storage. A chart upgrade that replaces the StatefulSet or changes
volume names can affect stored Vault data.

TLS changes need a client plan. Rotating the server certificate with the same
CA is different from rotating the CA. Rotating the CA means every client that
trusts the old CA needs the new CA before strict TLS checks will succeed. Adding
or changing exposure hostnames may require adding certificate SANs.

Storage changes need a migration plan. Changing from file storage to another
backend, changing `/vault/data`, resizing or replacing the PVC, or switching to
HA should be designed as a Vault storage migration, not a Helm value cleanup.
Back up first, document the restore path, and test status checks before and
after.

Auth and policy automation should be added deliberately. If this repo later
manages Vault policies with a Vault provider, that provider will need a way to
authenticate to Vault during preview/apply. Do not solve that by checking in
root tokens or broad long-lived tokens. Prefer a narrow bootstrap/admin
credential path whose handling is documented outside this public page.

Service exposure changes should preserve the private-by-default stance unless
there is a clear reason to do otherwise. A public route to Vault raises the
standard for network policy, authentication, TLS hostname coverage, monitoring,
and incident response.

Finally, keep the difference between Pulumi success and Vault success visible
in reviews. A clean preview means the infrastructure plan is coherent. It does
not prove that Vault is initialized, unsealed, reachable through every client
path, or ready for an application to depend on it. After an apply, the useful
follow-up checks are Kubernetes shape, TLS verification, `vault status`, and a
non-root auth/policy test for any affected consumer.
