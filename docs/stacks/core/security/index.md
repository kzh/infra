# Security Services

Security services are the cluster's trust and secret-management layer. They do
not only store sensitive bytes. They decide which controllers can issue trust
material, which workloads can consume it, how long that material remains valid,
and what evidence is safe to use when something fails.

This area currently centers on two stacks:

- [cert-manager](/stacks/core/security/cert-manager), the controller that turns
  certificate requests into Kubernetes TLS Secrets and renews them before
  expiry.
- [Vault](/stacks/core/security/vault), the in-cluster runtime secrets service
  whose usability depends on initialization, seal state, TLS, policies, auth
  methods, and storage.

Treat this page as the operating guide for the security layer as a whole. The
child pages describe the individual stacks. This page describes how to reason
across them without exposing the material they protect.

## The Security Layers

There are several secret and trust layers in this repo, and they are not
interchangeable.

Pulumi and ESC config are deploy-time infrastructure inputs. Use them for values
needed to create or configure infrastructure. Keep them secret through Pulumi's
secret mechanisms and ESC, and do not copy resolved values into docs, commit
messages, PRs, or chat.

Kubernetes Secrets are workload-delivery objects. A Secret can contain a TLS key,
a generated password, a service token, or a config file consumed by a pod. The
fact that a value is in a Kubernetes Secret does not make it safe to print. It
only means Kubernetes can mount or project it for a workload.

cert-manager owns certificate lifecycle. A `Certificate` is the request, an
`Issuer` or `ClusterIssuer` is the authority or signing path, and the resulting
Secret is the deliverable. If the Secret is wrong or stale, the durable fix is
usually in the `Certificate`, issuer, DNS/challenge path, or consuming workload
reference, not in the Secret data.

Vault is a runtime secret service. It is useful when a human or running service
needs a central API, policy-controlled access, dynamic credentials, or an
operational secret workflow after the infrastructure exists. It still needs a
clear auth method, policy, secret engine, rotation model, recovery model, and
client trust path.

Good security work starts by naming the layer that owns the value. A TLS serving
key generated by cert-manager, a Vault root token, a Pulumi secret config value,
and an application password mounted from a Kubernetes Secret all have different
owners and different recovery stories.

## Trust Material

Trust material is anything that lets one system decide whether another system is
who it claims to be. In this repo that usually means CAs, server certificates,
private keys, issuer configuration, Vault TLS material, and the Kubernetes
objects that bind those things together.

The sensitive part is not only the private key. The surrounding metadata can also
matter: DNS names, service names, issuer names, auth method names, secret path
shapes, and policy names can reveal internal design. Those details are often
fine to inspect locally, but they should be summarized carefully when reporting
outside the terminal.

When you work on trust material, keep these boundaries clear:

- The authority signs or issues material. For cert-manager that is an issuer or
  ACME flow. For Vault's local TLS, it may be generated by Pulumi and stored as a
  Kubernetes Secret.
- The deliverable is what the workload consumes. For Kubernetes TLS this is
  usually a Secret of type `kubernetes.io/tls`.
- The consumer is the ingress, service, pod, controller, CLI, or browser that
  must trust and use the material.
- The proof is observable behavior that does not require printing the material:
  ready conditions, Secret existence and type, endpoint readiness, certificate
  expiry metadata, and a successful TLS connection from the intended client
  context.

Do not rotate a CA or issuer because one pod is failing until you have shown the
failure is actually a trust problem. A sealed Vault server, a missing Service
endpoint, a stale ingress reference, and an untrusted CA can all look like
"secrets are broken" from the application side.

## TLS Lifecycle

TLS has a lifecycle, not a single configuration point:

```text
issuer or CA exists
certificate request is created
challenge or signing succeeds
TLS Secret is written
consumer references the Secret
consumer reloads or restarts if needed
certificate is renewed before expiry
old material ages out safely
```

Debug the lifecycle in that order. A missing Secret, an expired certificate, or a
browser warning is only the symptom. The cause may be an issuer that is not
ready, a DNS challenge that cannot complete, a webhook that is unavailable, a
Secret reference in the wrong namespace, or a consumer that has not reloaded
after renewal.

Safe inspection commands prefer status, events, and object relationships over
secret payloads:

```bash
kubectl get certificate,certificaterequest,issuer,clusterissuer -A
kubectl describe certificate -n <namespace> <certificate>
kubectl describe certificaterequest -n <namespace> <request>
kubectl get events -n <namespace> --sort-by=.lastTimestamp
kubectl get secret -n <namespace> <tls-secret> -o jsonpath='{.type}{"\n"}'
```

For a serving endpoint, prove the consumer path without dumping keys:

```bash
kubectl get ingress -n <namespace> <name>
kubectl get svc,endpoints -n <namespace>
curl -fsSIk https://<host>
```

If you need to inspect certificate dates or subject names, do it locally and
share only the relevant summary. Avoid pasting full PEM blocks, private keys, or
base64 Secret data into an issue, PR, doc, or chat transcript.

## Secrets Lifecycle

Secrets also have a lifecycle:

```text
source of truth is chosen
secret is created or synced
consumer receives it
consumer proves it can authenticate
rotation path is known
old value is removed or loses access
recovery path is tested before an emergency
```

The source of truth matters more than the storage object you happen to see while
debugging. If Pulumi generated a password, fix the Pulumi program or config. If
cert-manager wrote a TLS Secret, fix the certificate or issuer. If Vault is the
runtime source, fix the policy, auth method, engine, or client integration. If a
Helm chart rendered the Secret, fix the chart values.

Manual Secret edits are emergency tools, not normal operations. They are easy to
lose on the next reconciliation and difficult to audit. Before editing live
secret material, identify what controller owns it and what will happen during
the next sync, restart, renewal, or apply.

Useful non-revealing checks:

```bash
kubectl get secret -n <namespace> <name>
kubectl describe secret -n <namespace> <name>
kubectl get pods -n <namespace> -o wide
kubectl describe pod -n <namespace> <pod>
```

`kubectl describe secret` shows metadata, type, and key names, not decoded
values. Key names can still be sensitive in some contexts, so summarize them if
they reveal private systems.

For Vault, separate infrastructure health from application readiness:

```bash
kubectl get pods,svc,endpoints,secrets -n vault
kubectl logs -n vault -l app.kubernetes.io/name=vault --tail=200
vault status
```

Only run Vault CLI commands from a trusted environment with the right token and
CA trust. Do not paste root tokens, unseal keys, client tokens, or secret values
into docs or chat. If the task is to show someone how to retrieve a value, show
the local command and explain what the value is for, not the value itself.

## Rotation Thinking

A rotation plan is more than "make a new secret." It needs to answer five
questions.

What consumes the value? List the workload, namespace, chart value, environment
variable, mounted file, ingress, CLI, or human process that uses it.

Can the consumer accept two values during overlap? Some systems can trust old
and new CAs at the same time or accept two tokens during migration. Others need
a narrow cutover window.

What causes the consumer to reload? Some controllers watch Secret changes.
Others need a deployment rollout, pod restart, config reload endpoint, or manual
operator action.

How do you prove the new value works? Use a safe behavioral check: ready
conditions, endpoint health, authenticated request success, TLS handshake
success, or Vault policy read success from the intended identity.

How do you retire the old value? Remove old trust roots, revoke old tokens,
delete obsolete Kubernetes Secrets, or let cert-manager renewal and garbage
collection take over when that is the intended owner.

For TLS, CA rotation is the highest-risk case because clients may have pinned or
cached trust. Server certificate renewal is lower risk if the issuer and
consumer references remain stable. For Vault, root tokens and unseal keys are
operational recovery material, not application integration material. Application
access should flow through policies and auth methods that can be rotated without
resetting the service.

## Redaction Rules

Security debugging often creates evidence that is useful locally and unsafe to
share verbatim. Redact before writing anything durable.

Never include these values in docs, final messages, commit messages, PR text, or
shared logs:

- private keys, PEM blocks, root tokens, unseal keys, API tokens, bearer tokens,
  passwords, cookies, kubeconfig contents, decrypted Pulumi secret values, or
  decoded Kubernetes Secret data
- full Secret manifests when they include `data`, `stringData`, annotations from
  secret-sync tools, or token-like labels
- command output produced with `--show-secrets`, decoded base64 values, or
  verbose client traces containing authorization headers

Usually safe to share in summarized form:

- object kind, namespace, and non-sensitive name
- readiness condition type and status
- certificate expiry date and the fact that a Secret exists
- whether a Vault server is initialized, sealed, or reachable
- whether an issuer, webhook, Service, endpoint, or pod is healthy

When in doubt, report the shape instead of the value:

```text
The Certificate is Ready=False because the issuer is not ready.
The TLS Secret exists and has type kubernetes.io/tls.
Vault is initialized but sealed.
The consumer references a Secret name that does not exist in its namespace.
```

This is enough to guide the next step without leaking protected material.

## Debugging Without Leaking

Use a map-first workflow. Start with references and ownership, then inspect
status, then test behavior.

For cert-manager:

```bash
kubectl get certificate,issuer,clusterissuer -A
kubectl describe certificate -n <namespace> <certificate>
kubectl describe issuer -n <namespace> <issuer>
kubectl describe clusterissuer <clusterissuer>
kubectl get pods,svc,endpoints -n <cert-manager-namespace>
```

Look for `Ready` conditions, events, challenge failures, webhook availability,
and whether the expected Secret name appears in the same namespace as the
consumer.

For Vault:

```bash
kubectl get pods,svc,endpoints -n vault
kubectl logs -n vault -l app.kubernetes.io/name=vault --tail=200
vault status
```

Classify the failure before changing anything:

- TLS trust failure: the client does not trust the CA, uses the wrong host name,
  or reaches a server with a different certificate than expected.
- Seal-state failure: Vault is running but sealed.
- Storage failure: Vault cannot read or write its backend.
- Policy/auth failure: the server is healthy, but the identity does not have the
  needed path access.
- Network exposure failure: the client cannot reach the Service or endpoint.

For workloads consuming Secrets:

```bash
kubectl describe pod -n <namespace> <pod>
kubectl get deployment,statefulset,daemonset -n <namespace>
kubectl get events -n <namespace> --sort-by=.lastTimestamp
```

Look for missing Secret references, mount failures, restart loops, stale
rollouts, and application logs that identify the failing dependency without
printing the secret. If logs contain secret material, keep the raw logs local and
summarize the failure.

## Change Safety

For code or chart changes in security stacks, use the repo wrappers and stop at
preview unless an apply was explicitly requested:

```bash
just sync pulumi/core/security/cert-manager
just sync pulumi/core/security/vault
just check-python
just lint
git diff --check
just preview pulumi/core/security/cert-manager stack=mx
just preview pulumi/core/security/vault stack=mx
```

Before changing a security stack, write down the expected consumer impact. A
cert-manager chart upgrade can affect every `Certificate` consumer through CRDs,
webhooks, and controller behavior. A Vault TLS, storage, or chart change can
affect every client that authenticates to it or trusts its serving certificate.

After a change, verify the service and at least one real consumer path. For
cert-manager, that means controller pods, webhook endpoints, issuer readiness,
and an existing certificate. For Vault, that means pod readiness, service
endpoints, `vault status`, TLS behavior, and one policy-scoped access path if a
runtime integration was changed.

The goal is not to make security infrastructure dramatic. The goal is to keep
ownership clear, rotate deliberately, debug from evidence, and avoid turning
protected values into artifacts that outlive the incident.
