# Storage

This repo's storage area is the object-storage layer for the data platform.
The concrete implementation today is [RustFS](/stacks/data/storage/rustfs), an
S3-compatible service managed by `pulumi/data/storage/rustfs`. The important
thing to keep straight is ownership: RustFS provides the object-store service,
endpoint, credentials, and persistent backing volume. Consumer stacks own the
buckets and layouts they need.

Object storage is not a filesystem with directories. The durable contract is a
bucket name, an object key, object metadata, an endpoint, credentials, and the
client behavior expected by the application. Prefixes such as `warehouse/` are
conventions inside object keys, not directories with POSIX semantics. That
distinction matters when debugging tools like MLflow and Trino, because a pod
can be healthy while the bucket, endpoint, credentials, or table metadata it
depends on is wrong.

Treat stack outputs as APIs. The RustFS stack exports the namespace, console
hostname, S3 hostname, and generated credentials. MLflow and Trino consume those
outputs through `StackReference`s and materialize them into Kubernetes Secrets
inside their own namespaces. Do not copy secret values into source files, docs,
commit messages, or issue text.

## The Storage Plane

RustFS is installed as a standalone, single-replica object store in the
`rustfs` namespace. It uses the RustFS Helm chart, the `local-path` storage
class by default, a private console ingress, and a dedicated Tailscale-exposed
S3 API Service. The chart does not expose all of the Service annotations this
repo needs, so the S3 Service is created explicitly by Pulumi.

These endpoints have different audiences:

```text
Console:  https://rustfs
S3 API:   http://rustfs-s3:9000 from the tailnet
S3 API:   http://rustfs-s3.rustfs.svc.cluster.local:9000 from Kubernetes
```

The console is for humans. Applications should use the S3 API. If the console
loads but artifacts or Iceberg writes fail, debug the S3 endpoint, bucket,
credentials, and client configuration before changing ingress. If S3 clients
work but the console does not, debug the console ingress separately.

Read endpoint coordinates from outputs instead of hard-coding them in client
setup:

```bash
cd pulumi/data/storage/rustfs

NS="$(pulumi stack output --stack mx namespace)"
S3_HOST="$(pulumi stack output --stack mx s3_hostname)"
CONSOLE_HOST="$(pulumi stack output --stack mx console_hostname)"

printf 'Console: https://%s\n' "$CONSOLE_HOST"
printf 'S3 from tailnet: http://%s:9000\n' "$S3_HOST"
printf 'S3 from cluster: http://%s.%s.svc.cluster.local:9000\n' "$S3_HOST" "$NS"
```

## Buckets And Owners

The RustFS stack does not define application buckets. That is deliberate. A
bucket should be created by the stack that owns the data contract, because the
consumer knows the name, layout, lifecycle, and verification workflow.

The current repo-owned buckets are:

| Bucket | Owner stack | Creation path | Primary client | Verification |
| --- | --- | --- | --- | --- |
| `mlflow` | `pulumi/data/analytics/mlflow` | Job `mlflow-create-bucket` | MLflow artifact proxy | Log an MLflow run with an artifact and read it in the UI |
| `trino-iceberg` | `pulumi/data/analytics/trino` | Job `trino-iceberg-bucket` | Trino Iceberg catalog | Create, insert into, and query an Iceberg table |

For new buckets, add a clear owner. Prefer a bootstrap Job or equivalent
resource in the consumer stack, make it idempotent, and wire it to the same
RustFS endpoint and credentials the application will use. A manual bucket can
be useful for a short investigation, but it should either be deleted afterward
or documented with an owner and expected retention.

Do not rename or delete buckets casually. A bucket name is an application
contract. Pulumi may show a small change while the data consequence is large:
MLflow may keep running with missing artifacts, or Trino may keep serving
non-Iceberg catalogs while the Iceberg warehouse is unreachable.

## Credentials

RustFS credentials are generated by the RustFS stack and exported as Pulumi
secret outputs:

```bash
cd pulumi/data/storage/rustfs

pulumi stack output --stack mx access_key
pulumi stack output --stack mx secret_key
```

Those commands show that the outputs exist. Retrieving the actual values
requires `--show-secrets`; only do that in a local shell when configuring a
client, and avoid echoing the values back to the terminal.

Consumers should not duplicate credential literals. MLflow creates the Secret
`mlflow-artifacts-s3` with `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY`.
Trino creates the Secret `trino-catalog-credentials` with
`TRINO_S3_ACCESS_KEY` and `TRINO_S3_SECRET_KEY`. Both are populated from the
RustFS stack outputs. If the RustFS keys rotate, those consumer Secrets and the
pods that read them need to reconcile before the platform is healthy.

When debugging, confirm secret shape rather than printing secret values:

```bash
kubectl describe secret -n mlflow mlflow-artifacts-s3
kubectl describe secret -n trino trino-catalog-credentials
kubectl describe pod -n mlflow -l app=mlflow
kubectl describe pod -n trino -l app.kubernetes.io/name=trino
```

The question is whether the Secret exists, whether the pod references it, and
whether the expected key names are present. The key values themselves should
stay out of logs and chat.

## Client Patterns

S3-compatible clients need the same few pieces: endpoint URL, access key,
secret key, region, and sometimes path-style addressing. The repo uses
`us-east-1` as the default region for MLflow and Trino. Trino explicitly sets
`s3.path-style-access=true` for RustFS.

For a direct CLI check from a trusted machine on the tailnet:

```bash
cd pulumi/data/storage/rustfs

HOST="$(pulumi stack output --stack mx s3_hostname)"
ACCESS_KEY="$(pulumi stack output --stack mx --show-secrets access_key)"
SECRET_KEY="$(pulumi stack output --stack mx --show-secrets secret_key)"

AWS_ACCESS_KEY_ID="$ACCESS_KEY" \
AWS_SECRET_ACCESS_KEY="$SECRET_KEY" \
AWS_DEFAULT_REGION="us-east-1" \
aws --endpoint-url "http://$HOST:9000" s3 ls
```

For a reversible write/read check, use a temporary bucket name that cannot be
mistaken for an application bucket:

```bash
BUCKET="docs-storage-verification"
KEY="rustfs-storage-verification.txt"
OBJECT="/tmp/$KEY"

printf 'rustfs storage verification\n' > "$OBJECT"

AWS_ACCESS_KEY_ID="$ACCESS_KEY" \
AWS_SECRET_ACCESS_KEY="$SECRET_KEY" \
AWS_DEFAULT_REGION="us-east-1" \
aws --endpoint-url "http://$HOST:9000" s3 mb "s3://$BUCKET"

AWS_ACCESS_KEY_ID="$ACCESS_KEY" \
AWS_SECRET_ACCESS_KEY="$SECRET_KEY" \
AWS_DEFAULT_REGION="us-east-1" \
aws --endpoint-url "http://$HOST:9000" s3 cp "$OBJECT" "s3://$BUCKET/$KEY"

AWS_ACCESS_KEY_ID="$ACCESS_KEY" \
AWS_SECRET_ACCESS_KEY="$SECRET_KEY" \
AWS_DEFAULT_REGION="us-east-1" \
aws --endpoint-url "http://$HOST:9000" s3 cp "s3://$BUCKET/$KEY" -

AWS_ACCESS_KEY_ID="$ACCESS_KEY" \
AWS_SECRET_ACCESS_KEY="$SECRET_KEY" \
AWS_DEFAULT_REGION="us-east-1" \
aws --endpoint-url "http://$HOST:9000" s3 rm "s3://$BUCKET/$KEY"

AWS_ACCESS_KEY_ID="$ACCESS_KEY" \
AWS_SECRET_ACCESS_KEY="$SECRET_KEY" \
AWS_DEFAULT_REGION="us-east-1" \
aws --endpoint-url "http://$HOST:9000" s3 rb "s3://$BUCKET"
```

For Python clients, keep the endpoint and credentials in the environment or a
Kubernetes Secret. Do not check them into code:

```python
import os

import boto3

s3 = boto3.client(
    "s3",
    endpoint_url=os.environ["S3_ENDPOINT_URL"],
    aws_access_key_id=os.environ["AWS_ACCESS_KEY_ID"],
    aws_secret_access_key=os.environ["AWS_SECRET_ACCESS_KEY"],
    region_name=os.environ.get("AWS_DEFAULT_REGION", "us-east-1"),
)

print([bucket["Name"] for bucket in s3.list_buckets()["Buckets"]])
```

If the client is running inside Kubernetes, use the cluster DNS endpoint. If it
is running from the laptop, use the Tailscale hostname. Do not mix these up
while debugging; a working in-cluster endpoint does not prove that the tailnet
path works, and a working tailnet path does not prove that a pod has the right
DNS, Secret, or network route.

## Consumers

MLflow stores run metadata in PostgreSQL and artifact bytes in RustFS. The
MLflow stack consumes the RustFS S3 hostname, namespace, access key, and secret
key. It builds an endpoint of the form
`http://rustfs-s3.rustfs.svc.cluster.local:9000`, creates the `mlflow` bucket
through `mlflow-create-bucket`, and configures proxied artifact storage. That
last piece matters: normal MLflow clients should log artifacts through the
MLflow API, not by copying RustFS credentials into notebooks.

A useful MLflow object-storage check is not "is the pod ready?" It is:

```bash
cd pulumi/data/analytics/mlflow
NS="$(pulumi stack output --stack mx namespace)"

kubectl logs -n "$NS" job/mlflow-create-bucket --tail=100
kubectl logs -n "$NS" deploy/mlflow --tail=200
```

Then log a run with an artifact through the MLflow client and confirm the run,
metric, parameter, and artifact appear in the UI. If the run appears but the
artifact does not, the database path is working and the likely fault is the S3
endpoint, bucket, credentials, or artifact proxy configuration.

Trino stores Iceberg table metadata in PostgreSQL and table files in RustFS.
The Trino stack creates the `trino-iceberg` bucket, configures the Iceberg
catalog with `s3://trino-iceberg/warehouse`, enables native S3 support, uses
path-style access, and reads S3 keys from `trino-catalog-credentials`.

The useful Trino storage check is an Iceberg round trip:

```sql
show schemas from iceberg;

create schema if not exists iceberg.docs_verification;

create table if not exists iceberg.docs_verification.storage_roundtrip (
    id bigint,
    note varchar
);

insert into iceberg.docs_verification.storage_roundtrip
values (1, 'object storage round trip');

select * from iceberg.docs_verification.storage_roundtrip;
```

That verifies Trino, RustFS, the Iceberg bucket, S3 credentials, the warehouse
prefix, and the JDBC catalog metadata path together. If `tpch` queries work but
Iceberg writes fail, Trino is not wholly down; focus on RustFS, the bucket job,
S3 credentials, and the Iceberg JDBC catalog database.

Spark now uses this same RustFS-backed Iceberg storage plane through its
`trino_iceberg` catalog. Spark and Trino still use different user-facing catalog
names, but the backing contract is shared:

```text
Spark: trino_iceberg.<schema>.<table>
Trino: iceberg.<schema>.<table>
```

If that cross-engine view breaks, check both halves of the Iceberg contract:
RustFS objects under `s3://trino-iceberg/warehouse` and PostgreSQL JDBC catalog
metadata named `trino_iceberg`.

JupyterHub, workflow jobs, and batch jobs may be clients of RustFS, MLflow, or
Trino depending on the job. Prefer the highest-level API that preserves the
right semantics. Use MLflow for experiment artifacts, Trino for Iceberg tables,
and direct S3 access for raw objects that truly belong as objects.

## Before Changing Storage

Start by finding consumers. For the current repo, the important code paths are
RustFS itself plus MLflow and Trino:

```bash
rg -n "rustfs|s3_hostname|access_key|secret_key|MLFLOW_S3|icebergBucket|trino-iceberg" \
  pulumi docs \
  --glob '!**/uv.lock' \
  --glob '!pulumi/lib/**'
```

Then classify the change:

```text
Endpoint change:        preview RustFS and every consumer that builds endpoint URLs.
Credential rotation:    preview RustFS, MLflow, and Trino; plan pod reconciliation.
Bucket rename:          treat as a data migration, not a cosmetic rename.
Storage class change:   treat as a persistence and data-movement change.
Chart upgrade:          check service selectors, ports, PVCs, and client behavior.
Consumer bucket change: preview the consumer and run that consumer's workflow test.
```

Run cheap repo checks and a targeted preview. Do not apply from this workflow
unless the user explicitly requested an apply:

```bash
just sync pulumi/data/storage/rustfs
just check-python
just lint
git diff --check
just preview pulumi/data/storage/rustfs stack=mx
```

If a RustFS output or credential contract changed, also preview consumers:

```bash
just preview pulumi/data/analytics/mlflow stack=mx
just preview pulumi/data/analytics/trino stack=mx
```

Pulumi preview tells you what Kubernetes will change. It does not prove that
existing clients can still write and read object data. Keep the preview, but do
not stop there for storage work.

## Storage-Change Verification

After an approved apply, verify in layers. First confirm the RustFS Kubernetes
objects are present and the S3 Service has endpoints:

```bash
cd pulumi/data/storage/rustfs
NS="$(pulumi stack output --stack mx namespace)"

kubectl get pods,svc,ingress,pvc,secrets -n "$NS"
kubectl get endpoints -n "$NS" rustfs-s3
kubectl logs -n "$NS" -l app.kubernetes.io/name=rustfs --tail=200
```

Then do a direct S3 write/read/delete using a temporary bucket or temporary
object. That proves the API path, credentials, and object persistence are
usable from the client location you tested.

Finally verify the consumers that make the storage valuable:

```text
MLflow: log a run with an artifact and open the artifact through the UI.
Trino: create or query an Iceberg table and confirm reads after writes.
Existing buckets: list the expected buckets without exposing object contents in chat.
Storage migrations: verify old data paths before cleaning up any old bucket or PVC.
```

A healthy RustFS pod is only the first layer. The storage plane is healthy when
RustFS accepts S3 traffic, the expected buckets exist, credentials work through
the consumer Secrets, and the real clients can read and write through their
normal APIs.

## Failure Guide

When the console works but S3 clients fail, check `rustfs-s3`, port `9000`,
path-style behavior, the endpoint URL, and the credentials used by the client.

When direct S3 works but MLflow artifacts fail, check `mlflow-create-bucket`,
`mlflow-artifacts-s3`, `MLFLOW_S3_ENDPOINT_URL`, and whether the client is
logging through the intended tracking server.

When Trino works for `tpch` but Iceberg fails, check `trino-iceberg-bucket`,
`trino-catalog-credentials`, `spark-iceberg-credentials`, the `trino_iceberg`
JDBC catalog database, the warehouse URI, and RustFS access from the Trino and
Spark pods.

When a Service has no endpoints, compare the Service selector with pod labels
before changing client configuration. Traffic cannot reach a healthy pod if the
Service selector no longer matches it.

When credentials rotate, expect consumer reconciliation. A Secret changing does
not guarantee that a running process has re-read it. Preview and verify the
RustFS stack and each consumer that reads the credentials.

When changing object layouts, avoid manual edits under Iceberg warehouses.
Iceberg object files are tied to metadata and snapshots. Use the table engine to
create, rewrite, expire, or drop table data so metadata and object data stay in
sync.
