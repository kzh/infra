# WordPress

Source: `pulumi/apps/wordpress`

WordPress in this repository is a conventional PHP CMS deployed directly onto
Kubernetes. There is no Helm chart wrapping it and no custom application image
in the stack today. Pulumi creates the Kubernetes resources, the MySQL backing
database, the persistent WordPress volume, and the private route into the site.
Once the site is running, WordPress itself owns the editorial model: pages,
posts, media, menus, users, comments, plugin settings, theme settings, and most
day-to-day site configuration are stored through the WordPress application.

The important mental model is that WordPress is not only a container. It is a
PHP codebase plus a database plus a writable filesystem. The database contains
content and configuration. The filesystem contains uploaded media and, in this
stack, the persisted WordPress application tree under `/var/www/html`. A
healthy pod without the right database is not the site. A database dump without
the matching uploads, themes, and plugins is not the site either.

## What Pulumi Builds

The Pulumi project is a Python 3.12 stack using `pulumi-kubernetes`,
`pulumi-random`, and generated MySQL Operator CRD bindings from
`pulumi/lib/mysql_operator_crds`. The project entrypoint is
`pulumi/apps/wordpress/__main__.py`.

At a high level, the stack declares:

```text
Namespace:              wordpress
Application labels:     app=wordpress
WordPress Deployment:   wordpress
WordPress Service:      wordpress, ClusterIP, port 80
WordPress PVC:          wordpress-data, mounted at /var/www/html
Ingress:                wordpress, class tailscale
MySQL cluster:          wordpress-mysql, MySQL Operator InnoDBCluster
MySQL root Secret:      generated by Pulumi
WordPress DB Secret:    generated by Pulumi
DB init Job:            wordpress-db-init
Default database:       wordpress
Default database user:  wordpress
Default table prefix:   wp_
```

The current code defaults are:

```text
WordPress image:        wordpress:6.9.4-php8.3-apache
MySQL version:          9.7.0
MySQL instances:        1
MySQL routers:          1
MySQL storage:          20Gi
WordPress storage:      20Gi
```

Those values are code defaults, not a promise that every stack will always use
them. The project also accepts Pulumi config for namespace, hostname, public
URL, image versions, storage sizes, storage class, MySQL cluster name, database
name, database user, table prefix, and password lengths. Always inspect the live
stack outputs and config before assuming the deployed value.

Useful read-only orientation commands:

```bash
just sync pulumi/apps/wordpress

cd pulumi/apps/wordpress
pulumi stack output --stack mx namespace
pulumi stack output --stack mx hostname
pulumi stack output --stack mx url
pulumi stack output --stack mx wordpressImage
pulumi stack output --stack mx mysqlClusterName
pulumi stack output --stack mx mysqlHost
pulumi stack output --stack mx mysqlPort
```

From the repository root, the normal validation path for a code or config
change is:

```bash
just check-python
just lint
git diff --check
just preview pulumi/apps/wordpress stack=mx
```

Do not use `pulumi up`, `pulumi destroy`, or `just up` as an inspection tool.
An apply is a separate operation with live infrastructure effects.

## CMS Model

WordPress is a content management system built around mutable state. Editors
create pages and posts through the admin UI. Uploads land in the media library.
Themes decide how content is rendered. Plugins add behavior, admin screens,
shortcodes, blocks, background jobs, security controls, and integration state.
Most of those changes are not represented in this Git repository.

That is a feature and an operational cost. A UI-managed WordPress site is fast
to change, but Pulumi previews cannot tell you whether a plugin update changed a
page template or whether an editor changed a navigation menu. Pulumi owns the
infrastructure envelope. WordPress owns the CMS data inside that envelope.

For this stack, think of the contract this way:

```text
Pulumi owns:
  namespace, secrets, MySQL cluster, database bootstrap, PVC, Deployment,
  Service, Tailscale Ingress, probes, exported connection metadata

WordPress owns:
  users, roles, posts, pages, media metadata, theme selection, plugin settings,
  site options, and application-level migrations

The PVC owns:
  uploaded files, installed themes, installed plugins, and the persisted PHP
  application tree under /var/www/html
```

If the site should be reproducible entirely from Git, the current stack is only
the base. You would need to move themes, plugins, and any custom PHP into a
repo-backed image or another code-owned artifact. If the site is intentionally
managed through the WordPress admin UI, the infrastructure work is mostly about
protecting the database and filesystem, keeping upgrades safe, and making
recovery predictable.

## Database And Content Storage

The MySQL layer is an operator-managed `InnoDBCluster`. Pulumi creates a root
credential Secret and a separate WordPress application credential Secret. It then
declares the MySQL cluster with a data volume claim template and a router. The
WordPress application connects to the router service at the exported MySQL host
on port `3306`.

The database bootstrap is handled by the `wordpress-db-init` Job. The Job waits
for the MySQL router to respond, then creates the WordPress database and user,
sets the user password, grants privileges on the WordPress database, and flushes
privileges. The SQL is intentionally idempotent: it uses `CREATE DATABASE IF NOT
EXISTS`, `CREATE USER IF NOT EXISTS`, and `ALTER USER`.

The Job has a Pulumi-generated task annotation derived from the DB init revision,
database password, database name, database user, MySQL service host, and MySQL
version. If one of those inputs changes, Pulumi has a stable reason to replace
the Job and run the bootstrap logic again. The Job is also declared with
`delete_before_replace=True` and `replace_on_changes=["spec"]`, which avoids
same-name Job replacement friction.

The filesystem side is the `wordpress-data` PVC mounted at `/var/www/html`.
That mount point is broader than `wp-content`; it is the application document
root used by the official WordPress Apache image. As a result, the PVC can
contain uploaded media, installed plugins, installed themes, generated files,
and WordPress core files written during container bootstrap or in-app updates.

Backups, restores, and upgrades must treat these two state stores as a pair:

```text
MySQL database:
  posts, pages, users, roles, menus, comments, media metadata, options,
  plugin settings, theme settings, site URLs, scheduled tasks

wordpress-data PVC:
  wp-content/uploads, wp-content/plugins, wp-content/themes, generated files,
  and the persisted WordPress PHP tree under /var/www/html
```

If media library thumbnails exist in MySQL but the files are gone from the PVC,
the site will render broken images. If plugin files exist on the PVC but the DB
does not contain their settings and activation state, the app will not behave as
it did before. If the database is restored to one point in time and the PVC to
another, expect confusing application behavior.

## Admin Bootstrap

The DB init Job is not the WordPress installer. It creates MySQL plumbing only.
The first WordPress application bootstrap still happens through WordPress itself.
On a fresh install, open the stack URL after the MySQL cluster and DB init Job
are ready, then complete the WordPress browser installer. The admin account
created there is application data stored in MySQL.

There is no Pulumi output for a WordPress admin password, and this stack should
not grow one casually. Admin credentials are CMS-level secrets. Keep them in the
password manager or identity flow used for the site, not in source control,
commit messages, docs, or Pulumi plaintext config.

For routine admin access, use the WordPress login page at the exported URL:

```bash
cd pulumi/apps/wordpress
pulumi stack output --stack mx url
```

Then visit:

```text
<url>/wp-admin/
```

If the admin password is lost, reset it through WordPress-supported paths first:
email recovery, an authenticated admin session, or a controlled maintenance
operation with a known-good backup. Rotating the database password is not the
same thing as resetting a WordPress user password. The database password only
controls how the PHP application connects to MySQL.

The deployment sets `WORDPRESS_CONFIG_EXTRA` so `WP_HOME` and `WP_SITEURL` match
the Pulumi-computed public URL. That matters because WordPress stores and emits
absolute URLs in many places. If the site redirects to an unexpected host, check
the stack's `publicUrl`, `hostname`, and `tailscaleDomain` config before editing
files inside the running pod.

## Routing And URL Shape

Traffic enters through a Kubernetes Ingress with `ingress_class_name` set to
`tailscale`. The Ingress host is the configured `hostname`; by default that is
`wordpress`. It routes `/` to the `wordpress` ClusterIP Service on port `80`,
which selects pods with `app=wordpress` and forwards to the Apache container.

The externally reported WordPress URL is computed in code:

```text
publicUrl config if set
otherwise https://<hostname>.<tailscaleDomain> when tailscaleDomain is set
otherwise https://<hostname>
```

The same computed URL is written into WordPress config as `WP_HOME` and
`WP_SITEURL`. That keeps generated links, redirects, and admin URLs aligned with
the private Tailscale route.

When routing fails, separate the layers:

```bash
cd pulumi/apps/wordpress
NS="$(pulumi stack output --stack mx namespace)"

kubectl get ingress -n "$NS" wordpress
kubectl describe ingress -n "$NS" wordpress
kubectl get svc -n "$NS" wordpress
kubectl get endpoints -n "$NS" wordpress
kubectl get pods -n "$NS" -l app=wordpress
```

A `502` usually means the ingress layer reached a Service that did not have a
usable backend. Check Service endpoints and pod readiness before changing
WordPress settings. A redirect to the wrong URL is usually a WordPress URL
configuration issue. A browser certificate or tailnet access issue belongs to
the Tailscale routing layer.

## Operating The Running Site

Start with the stack outputs, then inspect Kubernetes objects in the namespace:

```bash
cd pulumi/apps/wordpress
NS="$(pulumi stack output --stack mx namespace)"

kubectl get pods,svc,ingress,pvc,jobs -n "$NS"
kubectl get innodbclusters -n "$NS"
kubectl logs -n "$NS" deploy/wordpress --tail=200
```

The DB init Job has `ttl_seconds_after_finished=3600`, so a successful Job may
disappear after an hour. If it is still present, it is useful to inspect:

```bash
kubectl logs -n "$NS" job/wordpress-db-init --tail=100
kubectl describe job -n "$NS" wordpress-db-init
```

For database readiness, inspect the MySQL custom resource and pods:

```bash
kubectl get innodbcluster -n "$NS"
kubectl describe innodbcluster -n "$NS" "$(pulumi stack output --stack mx mysqlClusterName)"
kubectl get pods -n "$NS"
```

For storage issues, inspect the PVC and the mounted filesystem from the
WordPress pod:

```bash
kubectl describe pvc -n "$NS" wordpress-data
kubectl exec -n "$NS" deploy/wordpress -- df -h /var/www/html
kubectl exec -n "$NS" deploy/wordpress -- ls -la /var/www/html/wp-content
```

Common symptoms and the first layer to check:

```text
Ingress returns 502:
  Service endpoints, pod readiness, Apache container logs

WordPress shows a database connection error:
  MySQL cluster readiness, router service, DB init Job, database Secret wiring

Uploads fail:
  PVC mount, available space, filesystem permissions, PHP upload limits

Admin redirects to the wrong host:
  publicUrl, hostname, tailscaleDomain, WP_HOME, WP_SITEURL

Plugin or theme update breaks the site:
  Apache/PHP logs, plugin directory, active theme, recent DB changes

Site is slow or intermittently unavailable:
  pod restarts, MySQL readiness, PVC latency, plugin behavior, background tasks
```

## Plugins And Themes

Plugins and themes are application code, even when installed through the admin
UI. In this stack, UI-installed plugins and themes are written onto the
`wordpress-data` PVC and their activation/configuration state is written into
MySQL. Pulumi will not show those changes in a preview.

That gives you two viable operating modes.

For a UI-managed site, it is acceptable for admins to install and update plugins
through WordPress, but every plugin/theme change should be treated as a real
application change. Take a backup first, change one thing at a time, and verify
the public page, admin UI, media library, and any forms or integrations that the
plugin owns.

For a reproducible site, do not rely on ad hoc UI installs. Build plugins,
themes, and any custom PHP into an image or another repo-owned artifact, then
make the WordPress Deployment consume that artifact. That is more work up
front, but it makes review, rollback, and rebuilds much more predictable.

Useful read-only checks:

```bash
kubectl exec -n "$NS" deploy/wordpress -- ls -1 /var/www/html/wp-content/plugins
kubectl exec -n "$NS" deploy/wordpress -- ls -1 /var/www/html/wp-content/themes
kubectl exec -n "$NS" deploy/wordpress -- du -sh /var/www/html/wp-content/uploads
```

If a plugin breaks the site badly enough that the admin UI is unavailable, a
common recovery move is to disable the plugin by moving its directory aside
during a maintenance window:

```bash
kubectl exec -n "$NS" deploy/wordpress -- \
  mv /var/www/html/wp-content/plugins/plugin-name \
     /var/www/html/wp-content/plugins/plugin-name.disabled
```

That is an emergency application operation, not an infrastructure fix. After the
site is reachable, decide whether to remove, replace, or reinstall the plugin
through the WordPress admin flow.

## Backup Strategy

This stack does not currently declare a repository-owned backup schedule for the
MySQL database or the `wordpress-data` PVC. Until one exists, backups are an
operational responsibility outside this Pulumi project. They should still be
tested and documented wherever the live backup process is managed.

A useful WordPress backup has three properties:

```text
Complete:
  It includes the MySQL database and the WordPress filesystem content.

Consistent:
  The database and filesystem represent the same maintenance window or point in
  time.

Restorable:
  Someone has restored it into a test or recovery target and confirmed the site
  works.
```

For a small internal site, the practical baseline is usually:

```text
1. Put the site in a quiet window or maintenance mode.
2. Dump the WordPress MySQL database.
3. Archive /var/www/html, or at minimum wp-content.
4. Store both artifacts together with the stack name and timestamp.
5. Verify that the archive contains uploads, themes, and plugins.
6. Periodically restore into a non-production target.
```

Example filesystem archive command:

```bash
cd pulumi/apps/wordpress
NS="$(pulumi stack output --stack mx namespace)"

kubectl exec -n "$NS" deploy/wordpress -- \
  tar -C /var/www/html -czf - wp-content > wordpress-wp-content.tgz
```

Archiving only `wp-content` is smaller and usually captures the unique files:
uploads, plugins, themes, and generated content. Archiving all of
`/var/www/html` is larger but can be useful before core or image upgrades
because this stack persists the full document root.

For database dumps, prefer a process that reads credentials from Kubernetes
Secrets or an approved secret store without printing them into shell history,
logs, docs, or chat. Use MySQL-native dump tooling or the MySQL Operator's
backup capability if it is added to the stack later. The important part is not
the exact dump command; it is that the dump and filesystem archive are taken as
a coordinated pair.

## Restore Strategy

Restores should be rehearsed before they are needed. A restore is successful
only when the running site behaves correctly, not when files have merely been
copied back.

A safe restore plan:

```text
1. Identify the target stack and namespace.
2. Take a fresh backup of the current broken state before overwriting anything.
3. Stop writes to the site with maintenance mode or a short downtime window.
4. Restore the MySQL database to the expected database name and table prefix.
5. Restore the matching WordPress filesystem archive to /var/www/html.
6. Bring the WordPress Deployment back to one replica if it was paused.
7. Open the site, log in to wp-admin, load existing media, and check plugins.
8. Run a Pulumi preview afterward to understand any infrastructure drift.
```

Do not restore only half the site unless you are deliberately repairing a narrow
problem. Restoring the database without the matching uploads can break media.
Restoring plugin files without the matching database state can leave plugins
missing settings or migrations.

Hostname changes deserve extra care. WordPress stores absolute URLs in options,
post content, and plugin data. The stack-level `WP_HOME` and `WP_SITEURL`
settings control the canonical runtime URL, but they do not rewrite every old
URL embedded in content. If a restore or migration changes hostnames, use a
WordPress-aware search-and-replace tool that understands serialized PHP data
instead of running broad manual SQL replacements.

## Safe Upgrades

There are three upgrade planes:

```text
WordPress/PHP image:
  Controlled by the Deployment image, defaulting to wordpress:6.9.4-php8.3-apache.

MySQL:
  Controlled by the MySQL Operator InnoDBCluster version and related client image.

Plugins/themes:
  Controlled inside WordPress unless this stack is later changed to use a
  repo-owned image or artifact.
```

Upgrade one plane at a time. A Pulumi preview can show Kubernetes replacements,
image changes, PVC changes, Service changes, and Ingress changes. It cannot
prove that WordPress database migrations, plugin migrations, or theme rendering
will work. The operational test still has to happen in the browser.

Before changing the WordPress image:

```text
1. Confirm recent backups for MySQL and the WordPress filesystem.
2. Check plugin and theme compatibility with the target WordPress/PHP version.
3. Preview the stack change.
4. Apply only during a window where rollback is possible.
5. After apply, check the home page, wp-admin, media library, editor, forms,
   login flow, and any plugin-owned pages.
```

Before changing MySQL:

```text
1. Read the MySQL Operator and MySQL version notes for upgrade constraints.
2. Confirm the MySQL PVCs and cluster identity are preserved.
3. Preview for replacements or immutable-field changes.
4. Verify the InnoDBCluster returns to ready before debugging WordPress.
5. Check the DB init Job only after the router and cluster are healthy.
```

Before changing plugins or themes:

```text
1. Back up database and filesystem together.
2. Update one plugin or theme at a time.
3. Keep the admin UI open in a known-good browser session.
4. Verify public pages and plugin-owned workflows.
5. If the site breaks, disable the changed plugin/theme first, then investigate.
```

Storage changes are also migrations. Growing a PVC is often supported by the
storage class, but shrinking usually is not. Changing `storageClassName` on an
existing PVC is generally not a small edit. Treat storage class changes as data
migrations with a backup and restore plan.

Stable names matter. Renaming the namespace, MySQL cluster, database, table
prefix, or PVC can strand state or force replacements. If a rename is required,
plan it as a migration and preserve state intentionally.

## Making Repo Changes

Most safe changes to this project start by reading the current stack state and
then making the smallest Pulumi edit that expresses the desired behavior.

Useful checks:

```bash
git status --short
just sync pulumi/apps/wordpress
just check-python
just lint
git diff --check
just preview pulumi/apps/wordpress stack=mx
```

When reviewing a WordPress change, pay attention to:

```text
Deployment image:
  Does it change WordPress or PHP behavior?

PVC:
  Does it preserve wordpress-data and avoid accidental replacement?

MySQL cluster:
  Does it preserve the cluster name, storage, and database identity?

Secrets:
  Are new values generated or read from secret config rather than committed?

DB init Job:
  Is it still idempotent, and will it rerun only when intended?

Ingress and URL config:
  Do hostname, publicUrl, WP_HOME, and WP_SITEURL still agree?
```

After an apply, the real acceptance test is end-to-end:

```text
1. The Tailscale URL loads.
2. wp-admin login works.
3. Existing pages and posts render.
4. Existing media files load.
5. Active plugins and the active theme are present.
6. A small draft edit can be saved.
7. No unexpected pod restarts appear after the change.
```

If those checks pass, Pulumi and WordPress agree closely enough for normal
operation. If they do not, debug by layer: route, pod, filesystem, database,
then WordPress application behavior.
