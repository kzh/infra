# MediaWiki

Source: `pulumi/apps/mediawiki`

MediaWiki looks like a website, but in this repository it should be operated as
a stateful application. The web container is replaceable. The wiki is not. The
wiki is the combination of database rows, uploaded files, generated
configuration, and credentials that have to keep agreeing with each other.

A page edit writes MediaWiki application state into MySQL. A file upload lands
on the `images` volume. The PHP process reads `LocalSettings.php` from a
Kubernetes Secret generated by Pulumi. The deployment is useful only when all of
those pieces line up.

That mental model is the most important thing about this stack. If the pod is
restarted, the wiki should survive. If the database or images volume is lost,
the wiki is damaged even if Kubernetes can create a fresh pod.

## What Pulumi Owns

The project under `pulumi/apps/mediawiki` creates the full application slice for
the wiki:

| Layer | Pulumi-managed objects |
| --- | --- |
| Namespace | `mediawiki` namespace by default |
| Credentials | generated MySQL root, MediaWiki DB, MediaWiki admin, secret key, and upgrade key material |
| Database | MySQL Operator `InnoDBCluster` plus a database/user initialization Job |
| App install | one-time/idempotent MediaWiki install Job |
| App updates | MediaWiki maintenance update Job |
| MySQL compatibility | SQL compatibility Job for tables that need to work with MySQL Group Replication |
| Runtime config | generated `LocalSettings.php` stored in a Secret |
| Runtime app | MediaWiki Deployment, ClusterIP Service, and Tailscale Ingress |
| Upload state | `mediawiki-images` PersistentVolumeClaim mounted at `/var/www/html/images` |
| Observability | Grafana dashboard ConfigMap from `dashboards/mediawiki-overview.json` |

The checked-in `mx` stack config currently describes this deployment shape:

```text
Namespace:          mediawiki
Ingress host:       wiki
MediaWiki image:    mediawiki:1.45.3
MySQL version:      9.7.0
MySQL cluster:      mediawiki-mysql
MySQL instances:    1
MySQL routers:      1
MySQL storage:      20Gi
Images storage:     20Gi
Database name:      mediawiki

Admin user:         Admin

```

Read the actual URL from the stack output instead of copying hostnames into
notes or screenshots. The Pulumi program builds the URL from the configured
host and Tailscale domain when that domain is present.

## What Belongs In The Wiki

MediaWiki is a good home for living, linked knowledge that benefits from page
history, quick edits, categories, templates, user accounts, and uploads. It is a
different tool than this VitePress documentation tree.

Use repository docs for procedures that should be reviewed with infrastructure
changes, such as stack ownership, recovery steps, and deployment constraints.
Use the wiki for knowledge that changes through the UI and is naturally
maintained by page editors. When a wiki page becomes operationally critical,
link to it from the relevant repo doc or mirror the durable procedure back into
version control.

The wiki has its own review and backup model. A merged Git commit does not back
up a wiki page, and a wiki edit does not go through code review. Choose the home
for a piece of knowledge based on that tradeoff.

## Opening The Wiki

Use Pulumi outputs for access details:

```bash
cd pulumi/apps/mediawiki

pulumi stack output --stack mx url
pulumi stack output --stack mx adminUser
pulumi stack output --stack mx localSettingsSecretName
```

Retrieve the generated admin password only in your own terminal:

```bash
pulumi stack output --stack mx --show-secrets adminPassword
```

The stable admin username is the `adminUser` output, currently `Admin` in the
checked-in `mx` config. The login page is:

```text
<url output>/index.php/Special:UserLogin
```

Do not paste the generated password into issues, docs, chat, screenshots, or PR
messages. If you need to hand off access, use a secret manager or rotate through
a deliberate process rather than copying the Pulumi secret value around.

## The State Planes

There are two durable state planes and one generated configuration plane:

```text
MySQL:       pages, revisions, users, permissions, metadata, sessions, jobs
Images PVC: uploaded files under /var/www/html/images
Secret:     generated LocalSettings.php and credentials mounted into pods/jobs
```

The Deployment can be recreated. The Service and Ingress can be recreated. The
Jobs can be recreated. Those are control-plane and runtime objects. The database
and uploaded files are the wiki.

That distinction should guide operations:

- A failing pod is usually a symptom, not the root state.
- A healthy pod does not prove schema migrations succeeded.
- An ingress response does not prove uploads are intact.
- A database backup without the images volume is an incomplete wiki backup.
- An images backup without a matching database backup can leave file metadata
  and files out of sync.

## MySQL Dependency

The stack uses the MySQL Operator CRDs through the repo-local generated Python
package `pulumi_mysql_operator_crds`. The app database is not a standalone
`Deployment`; it is an operator-managed `InnoDBCluster`.

The MediaWiki PHP container connects through the MySQL router service:

```text
<mysqlClusterName>.<namespace>.svc.cluster.local:3306
```

The compatibility Job connects to the first instance host because it needs to
make low-level table adjustments:

```text
<mysqlClusterName>-0.<mysqlClusterName>-instances.<namespace>.svc.cluster.local:3306
```

The database initialization Job waits for the router, creates the MediaWiki
database if needed, creates or updates the application database user, grants the
database privileges, and exits. The MediaWiki install Job then waits for the app
database using PHP `mysqli`. This matters because the official MediaWiki image
path used here is checked against `mysqli`, not `pdo_mysql`.

If the wiki is down, start with MySQL readiness before PHP debugging:

```bash
cd pulumi/apps/mediawiki

NS="$(pulumi stack output --stack mx namespace)"
MYSQL="$(pulumi stack output --stack mx mysqlClusterName)"

kubectl get innodbcluster -n "$NS" "$MYSQL"
kubectl get pods -n "$NS" -o wide
kubectl get svc,endpoints -n "$NS"
```

The `InnoDBCluster` has `pulumi.com/skipAwait` set. That keeps Pulumi from
blocking on operator readiness semantics that are better inspected through the
operator and Kubernetes state. It also means a completed Pulumi preview or apply
does not by itself prove MySQL is ready for MediaWiki traffic.

## Generated LocalSettings.php

`LocalSettings.php` is generated by the Pulumi program and stored in the
`mediawiki-local-settings` Secret. The Deployment and update Job mount that
Secret at:

```text
/var/www/html/LocalSettings.php
```

Do not edit `LocalSettings.php` inside a running pod. That edit is not durable,
and the next rollout will replace it. Change the Pulumi program or stack config,
preview the change, and then apply only when an apply has been explicitly
requested.

The generated config currently controls the core MediaWiki runtime settings:

- site name and namespace
- canonical server URL and script path
- MySQL host, database, user, password, prefix, and table options
- upload enablement
- language
- cache behavior
- secret key and upgrade key
- Vector skin loading
- HTTPS behavior

Some of those values are secrets. It is fine to inspect the Secret name or the
fact that a key exists. Avoid dumping the full Secret data into logs or
conversation. When the config shape itself is what matters, prefer reading the
Python renderer in `pulumi/apps/mediawiki/__main__.py` rather than printing the
live Secret.

The program computes task annotations from stable hashes. That is how a
meaningful config or Job input change causes Kubernetes to replace the right
Job or roll the app pod. The revision constants near the top of `__main__.py`
exist for the cases where the operational script changed but the normal inputs
would otherwise hash the same.

## Install And Maintenance Jobs

The stack uses Kubernetes Jobs for the parts of MediaWiki lifecycle that should
not be hidden inside the web Deployment.

`mediawiki-db-init` prepares MySQL. It waits for the router service, creates the
database, creates or updates the app database user, grants privileges, and
flushes privileges. It depends on the MySQL cluster and both the root and app DB
credential Secrets.

`mediawiki-install` is the first MediaWiki application bootstrap. It waits until
PHP can connect to the app database with `mysqli`. If the `site_stats` table is
already present, it treats the schema as installed and exits successfully. If
not, it runs `php maintenance/run.php install` with password files mounted from
Secrets.

`mediawiki-update` runs `php maintenance/run.php update --quick` with the
generated `LocalSettings.php` mounted. This is the database schema update step
for MediaWiki changes.

`mediawiki-db-compat` runs after the MediaWiki update. It adjusts known tables
so the MediaWiki schema is compatible with the MySQL Operator / Group
Replication setup. In the current source, it ensures the `searchindex` table is
InnoDB and adds invisible primary-key columns to selected tables when they exist
without a primary key. Treat this as part of the stack's database migration
story, not as incidental SQL.

The Jobs use `delete_before_replace=True` and `replace_on_changes=["spec"]`.
When their meaningful inputs change, Pulumi replaces the Job rather than trying
to mutate an old completed Job in place.

Finished Jobs are configured with a one-hour TTL, so inspect logs soon after an
install or upgrade:

```bash
cd pulumi/apps/mediawiki

NS="$(pulumi stack output --stack mx namespace)"

kubectl get jobs -n "$NS" -l app=mediawiki
kubectl logs -n "$NS" job/mediawiki-db-init --tail=200
kubectl logs -n "$NS" job/mediawiki-install --tail=200
kubectl logs -n "$NS" job/mediawiki-update --tail=200
kubectl logs -n "$NS" job/mediawiki-db-compat --tail=200
```

If a Job has already been removed by TTL, inspect the current Kubernetes events,
Pulumi state, and app/database health rather than assuming the old logs still
exist.

## Uploads And Files

Uploads are enabled in generated config with `$wgEnableUploads = true`. The
MediaWiki container mounts the `mediawiki-images` PVC at:

```text
/var/www/html/images
```

The pod security context sets `fsGroup` to `33`, matching the web server user in
the MediaWiki image. That is what lets the container write uploaded files to the
mounted volume.

Basic upload checks:

```bash
cd pulumi/apps/mediawiki

NS="$(pulumi stack output --stack mx namespace)"

kubectl get pvc -n "$NS" mediawiki-images
kubectl describe pvc -n "$NS" mediawiki-images
kubectl exec -n "$NS" deploy/mediawiki -- sh -lc 'id; ls -ld /var/www/html/images'
```

Do not delete `mediawiki-images` unless you are intentionally discarding all
uploaded files. Recreating the Deployment will not recreate those files. If
uploads fail after an image or storage change, check the PVC, mount path, and
permissions before changing MediaWiki application settings.

The Deployment currently runs one replica. Do not casually increase the replica
count. The upload volume is `ReadWriteOnce`, cache/session behavior is simple,
and scaling the web tier would need a real review of storage access, cache
behavior, and MediaWiki runtime assumptions.

## Observability

The project ships a Grafana dashboard ConfigMap labeled with
`grafana_dashboard=1`. The dashboard is intended to cover the basic operating
signals:

- MediaWiki replicas and restarts
- MySQL ready replicas and router replicas
- pod CPU and memory
- Tailscale ingress traffic
- persistent storage

For command-line inspection, start with:

```bash
cd pulumi/apps/mediawiki

NS="$(pulumi stack output --stack mx namespace)"

kubectl get deploy,svc,ingress,pvc,job -n "$NS" -l app=mediawiki
kubectl logs -n "$NS" deploy/mediawiki --tail=200
kubectl describe deploy -n "$NS" mediawiki
kubectl get events -n "$NS" --sort-by=.lastTimestamp
```

Treat `kubectl get secret` as a names-only inventory command. Do not use
`kubectl get secret -o yaml` or equivalent in copied diagnostics unless the
output has been reviewed and redacted.

## Backups

This Pulumi project provisions durable storage, but the current source does not
define a full MediaWiki backup and restore workflow. Until that exists in the
repo, assume backup work needs explicit operator attention.

A complete backup needs all of these pieces from the same logical point in
time:

- MySQL contents for the MediaWiki database
- uploaded files from `mediawiki-images`
- the Pulumi commit and non-secret stack configuration that produced the
  runtime shape
- a secret-safe record of where the credentials are managed

The database and images volume are linked. The database contains file metadata,
page references, users, and revision history. The PVC contains the uploaded file
bytes. A database dump without matching files restores pages that reference
missing uploads. A file copy without the matching database loses the metadata
that makes the files useful inside MediaWiki.

For a durable backup design, prefer one of these approaches:

- Add a repo-managed MySQL Operator backup resource if the operator version and
  storage backend support the desired target.
- Use a controlled dump Job that reads credentials from Kubernetes Secrets and
  writes to a secure backup location without printing secrets.
- Pair the database backup with a PVC snapshot or controlled file copy of
  `mediawiki-images`.

Do not treat an untested backup as complete. A real test restore should bring up
the database and images together in a disposable environment, run MediaWiki
maintenance updates, open the wiki, read pages, log in, and verify uploaded
files render.

## Safe Changes

Make durable changes in `pulumi/apps/mediawiki`, not by hand-editing live
Kubernetes objects. The stack is small enough that most changes should be
visible in one targeted preview.

Use the repository gates before asking for or running an apply:

```bash
just sync pulumi/apps/mediawiki
just check-python
just lint
git diff --check
just preview pulumi/apps/mediawiki stack=mx
```

Do not run `pulumi up`, `pulumi destroy`, or `just up` unless the user
explicitly asks for an apply or destructive action.

Common change types need different levels of care:

| Change | What to think about |
| --- | --- |
| MediaWiki image | Read release notes, preview, watch install/update/compat Jobs, verify `Special:Version`, page reads, edits, and uploads. |
| MySQL version | Confirm MySQL Operator support first, take backups, preview the CRD diff, and watch cluster readiness. |
| Hostname or public URL | Change config so Ingress and generated `LocalSettings.php` stay consistent. |
| Storage size | Expansion may be possible depending on the storage class. Shrinking is not a normal safe operation. |
| Upload behavior | Remember that config and PVC state both matter. Back up files before risky storage changes. |
| Admin credentials | The password is generated and exported as a Pulumi secret. Do not patch the Kubernetes Secret by hand as a long-term fix. Design rotation deliberately. |
| Generated config | Change `render_local_settings` or stack config, and rely on the Secret plus pod annotation flow to roll the app. |
| Maintenance SQL | Treat compatibility changes as schema migrations. Keep them idempotent and review them like database changes. |

After an approved apply, do not stop at "the Deployment rolled out." Verify the
wiki at the application level:

```bash
cd pulumi/apps/mediawiki

URL="$(pulumi stack output --stack mx url)"
NS="$(pulumi stack output --stack mx namespace)"

curl -kfsSI "$URL/"
curl -kfsSI "$URL/index.php/Special:UserLogin"
kubectl get jobs -n "$NS" -l app=mediawiki
kubectl logs -n "$NS" deploy/mediawiki --tail=100
```

Then use the UI for the checks that HTTP headers cannot prove: log in, read a
real page, make a small test edit if appropriate, and view or upload a file if
uploads are part of the change.

## Troubleshooting Patterns

If the ingress returns 502 or the URL does not load, check the Service endpoints
and pod readiness first:

```bash
NS="$(pulumi stack output --stack mx namespace)"

kubectl get ingress,svc,endpoints -n "$NS" -l app=mediawiki
kubectl get pods -n "$NS" -l app=mediawiki -o wide
kubectl describe pod -n "$NS" -l app=mediawiki,component=web
```

If the install Job fails, check MySQL readiness, database credentials wiring,
and the install Job logs. The install Job is designed to skip once the
`site_stats` table exists, so a successful "already exists; skipping install"
message is normal after the initial bootstrap.

If the update Job fails, inspect the generated config path, the MediaWiki image
version, and the maintenance command output. Schema-update failures are not
fixed by restarting the web Deployment.

If the compatibility Job fails, treat it as a database migration problem. Check
whether the target tables exist, whether MySQL is accepting writes on the
instance host, and whether the operator is still reconciling.

If uploads fail, check in this order:

1. `mediawiki-images` PVC is bound.
2. The Deployment mounted it at `/var/www/html/images`.
3. The mounted directory is writable by the container through `fsGroup`.
4. `$wgEnableUploads` is present in generated config.
5. The UI or PHP logs show an application-level upload error.

If login fails, confirm the username from `adminUser` and retrieve the password
locally from the Pulumi secret output. Do not reset credentials through a live
Secret patch unless the intent is temporary recovery and the follow-up is to
make the durable source of truth match.

If MediaWiki loads but pages error, check the Deployment logs and database
connectivity before changing ingress. Ingress can deliver traffic to a PHP app
that is still failing on schema, credentials, or generated config.
